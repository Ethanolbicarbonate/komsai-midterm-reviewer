SOFTWARE PROCESS

CIT 214 - Software Engineering

The Software Process

A software process prescribes how a software project progresses through various activities from inception to post-release maintenance.

A software project progresses through a series of activities, starting at its inception and continuing even beyond its release to customers. Typically, a project is organized into phases, each with a prescribed set of activities conducted during that phase.

A software process prescribes the interrelationships among the phases by expressing their order and frequency, as well as defining the deliverables of the project. It also specifies the criteria for moving from one phase to the next.

The Activities of Software Process

Most software process models include similar phases but differ in their order and frequency of execution.

Most software process models prescribe a similar set of phases and activities. The difference between models is the order and frequency of the phases. Some process models, like the waterfall, execute each phase only once while others, like the iterative models, cycle through multiple times.

Core Activities of Software Process

  • Inception
    Software product is conceived and defined
  • Planning
    Initial schedule, resources, and cost are determined
  • Requirements Analysis
    Specify what the application must do, answers "what?"
  • Design
    Specify the parts and how they fit, answers "how?"
  • Implementation
    Write code
  • Testing
    Execute the application with input test data
  • Maintenance
    Repair defects and add capability

1. Inception

Inception is the phase where initial product ideas are formulated and a vision of the software is conceived.

Whether it is a brand new product or an improvement to existing software, every project starts with an idea of what is to be built. Major functionality and project scope is defined. Target customers and market segments are identified. During inception, the goal is to gather very high level feedback.

Inception Example Scenario

As an example, suppose a company is considering building a video store application. A market analysis may be conducted to determine the existence of competing video store applications. A summary of their features and an analysis of their strengths and weaknesses is compiled. The commercial success of the proposed application is determined by identifying potential new customers and contacting them for information.

2. Planning

Planning is the phase where a plan is formulated to produce the application after a high-level idea is developed.

Once a high-level idea for the application is developed, a plan is formulated to produce it. Since this is still very early in the overall life cycle and detailed information is not yet available, the plan will be a rough estimate. A project plan that identifies the high-level activities, work items, schedule, and resources is developed. From this information a costs estimate can be developed.

If the cost is deemed too high, a reduction in features may be necessary. If the product release date is too late, more resources may be added.

Planning Deliverables

The result of this phase is typically captured in a Software Project Management Plan (SPMP). A plan is also developed for managing project artifacts such as technical specifications and source code. This is known as configuration management, and includes tasks such as tracking changes to artifacts and handling multiple versions.

Best Practice

Configuration management is planned at this early project stage, before the first project artifacts are generated.

3. Requirements Analysis

Requirements Analysis is the phase where detailed information regarding customer wants and needs, and problems the software is intended to solve are gathered.

Information gathered and documented is in much greater detail than it was in the inception phase. During this phase, specific product functions and features are defined along with requirements such as performance, reliability, and usability. Requirements must be generated in a form that is completely readable and understandable by customers.

High-level requirements in particular are typically expressed in ordinary English (or the local language). Various techniques are used to obtain this information, including customer interviews and brainstorming sessions.

Requirements Analysis Outcome

Requirements describe what the application is intended to accomplish and are specified in sufficient detail so they can be used as input for the subsequent design phase, which defines how the software will be built. The results of this phase are typically captured in a formal Software Requirements Specification (SRS) document, which serves as input to the next phase.

4. Software Design

Software Design is the phase that defines how the software will be constructed to satisfy the requirements. The internal structure of the software is defined.

The two main levels of software design are software architecture and detailed design.

Software Architecture

For example, the architecture for a video store application may consist of components such as user interface module, a problem domain module, and a database module.

Software architecture specifies how the software is broken into subsystems or modules and the software interfaces between them.

Detailed Design

Agile projects generally perform design, implementation, and much testing in an interweaved fashion rather than calling out design as a separate phase.

The detailed design would include details such as algorithms and data structures, as well as user interface design and database design.

The output of software design is specified in a Software Design Document (SDD) and is used as input to the implementation phase.

5. Implementation

Implementation is the phase that consists of programming, which is the translation of the software design developed in the previous phase to a programming language.

It also involves integration, the assembly of software parts. The output consists of program code that is ready to be tested for correctness.

Implementation in Agile Approaches

For agile techniques, design implementation and much testing are performed in tandem.

6. Testing

Testing is the phase where the code produced during the implementation phase is tested for correctness.

Testing Levels

Testing is performed in three levels:

  • First, individual modules are tested by developers.
  • Second, modules are integrated and tested to ensure that they interface properly.
  • Third, once all the modules had been integrated, the entire system is tested to ensure that it meets the user requirements.

System testing is typically conducted by an independent quality assurance (QA) team.

Customer Testing (Beta Testing)

Once system testing has been completed, several levels of customer testing are conducted. During beta testing, the software is as close to the final release as possible, and is given to part of the customer community with the understanding that they report defects they find. The goal is to uncover a set of problems that could only be discovered with this type of "real-world" testing.

Acceptance Testing

Once beta testing is complete, acceptance testing is conducted on the "final" release of software. Acceptance tests are comprised of a subset of system tests, and are conducted by either the customer or a representative of the customer to ensure that it meets the customer's criteria for release. Sometimes a round of acceptance testing is conducted prior to beta testing, to make sure that the system meets certain criteria before it is given to customers for beta testing.

7. Maintenance

Maintenance is the phase that begins after the software is officially released to customers.

During maintenance, modifications are made to the software that arise through one of the following:

  • The repair of software defects that are discovered during normal customer use of the system
  • Customer requests for enhancements
  • A desire to improve attributes of the system such as performance or reliability

Modifications to the software are bundled together, and new versions of the software with these modifications are released.

Software Process Examples

Below are examples illustrating each phase of the software process in the context of a video rental application.

Software Process Phase Examples

  • Inception: "An application is needed to keep track of video rentals"
  • Planning: "The project will take 12 months, require 10 people and cost $2M"
  • Requirements analysis: "The clerk shall enter video title, renter name and date rented. The system shall.."
  • Design: "classes Video Store, DVD, related by…"
  • Implementation: "class VideoStore(String name,..)..."
  • Testing: Run test case: Rent "The Matrix" on Oct 3
    Result: "The Matrix" due Oct 3, balance of $8 (correct)..
  • Maintenance:
    • Defect repair: "Application crashes when balance is $10 and attempt is made to rent 'Gone With the Wind'"
    • Enhancement: Allow searching by director
