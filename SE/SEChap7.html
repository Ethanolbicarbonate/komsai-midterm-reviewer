SOFTWARE PROCESS MODELS

CIT 214 - Software Engineering

What are Software Process Models?

Process models define the approach and workflow for developing software. Each model offers different advantages depending on project requirements, team size, and risk factors.

WATERFALL PROCESS MODEL

One of the oldest software process models. Despite many of its weaknesses, the waterfall process is still in widespread use today and is the basis for many other more effective processes.

Key Characteristics

The waterfall model follows a sequential approach with distinct phases. Each phase must be completed before moving to the next, with minimal overlap between phases.

How It Works

Organizations typically implement waterfall starting with requirements, excluding inception and planning phases. Each phase produces specific documentation before proceeding to the next phase. Entrance and exit criteria ensure successful phase completion.

Important Limitations

Advantages of the Waterfall Process Model

  • Simple and easy to use Phases are executed and completed serially, with specific entrance and exit criteria for moving between phases. Orderly execution of phases is easy to comprehend.
  • Practiced for many years and people have much experience with it The process is well understood, and many people are comfortable with its execution.
  • Easy to manage due to the rigidity of the model Each phase has specific deliverables and a review process.
  • Facilitates allocation of resources Distinct phases facilitate allocation of personnel with distinct skills.
  • Works well for smaller projects where requirements are very well understood It's necessary to add complexity of iteration if requirements are well known up front.

Disadvantages of the Waterfall Process Model

  • Requirements must be known up front It's difficult to imagine every detail in advance. Most projects start out with some uncertainty, and more details are learned as the project progresses.
  • Hard to estimate reliably To gain confidence in an estimate, there may be a need to design and implement parts, especially riskier ones. Estimate become more precise as the project progresses.
  • No feedback of system by stakeholders until after testing phase The process does not facilitate intermediate versions. Stakeholders often need reassurance of progress and confirmation that what is being developed meets requirements.
  • Major problems with system aren't discovered until late in process The testing phase is where these problems are found, but it leaves very little time for correction, resulting in potentially disastrous effects on project schedule and cost.
  • Lack of parallelism Each phase is executed to completion. Disjointed parts of the system could otherwise be completed in parallel.
  • Inefficient use of resources Team members can be idle while waiting for others to complete their dependent tasks or for phases to complete. Someone good at requirements analysis is not necessarily good at programming.

WATERFALL (unsummarized)

In practice, most organizations implement a waterfall process starting with the requirements phase, excluding the inception and planning phases. The waterfall executes in a sequential manner through each phase, concluding with maintenance.

The output from one phase is used as input for the next, implying that a phase is not started until the previous one has completed. It is accepted in waterfall processes that there is a small overlap between adjacent phases.

The waterfall process is characterized by documents generated by the time each phase is completed. These include requirements specification and design specification. There are usually entrance and exit criteria between phases to ensure that a phase is completed successfully before moving on.

In practice, an iterative relationship between successive phases is usually inevitable. A major limitation of the waterfall process is that the testing phase occurs at the end of the development cycle - the first time the system is tested as a whole.

Major issues such as timing, performance, storage, and so on can be discovered only then. Discovering and repairing defects so late in the development cycle can jeopardize the project schedule.

ITERATIVE AND INCREMENTAL MODEL

Core Concept

Software is more naturally developed in a cyclical manner, where a part of the system is developed and tested, feedback is gathered, and based on the feedback more of the system is developed.

An iterative process is typified by repeated execution of the waterfall phases, in whole or in part, resulting in a refinement of the requirements, design, and implementation.

An iterative process is incremental if each iteration is relatively small. At the conclusion of such an iteration, a piece of operational code is produced that supports a subset of the final product functionality and features.

Key Elements

Project artifacts evolve during each phase. Successive increments implement increasing functionality until the final product. Each iteration is self-contained with its own activities, objectives, and evaluation criteria. Iterations can be developed in parallel by separate teams.

Release Types

  • Proof of concept or feasibility study
    Demonstrates or investigates feasibility of particular aspects
  • Prototype
    Working version demonstrating high-risk capabilities
  • Internal release
    Used by the development team to ensure development is on track
  • External release
    Shipped to customers for evaluation

Best Practices

Testing each iteration as a self-contained project allows clear objectives and reduces complexity. Producing working software at each iteration helps monitor progress and gather feedback. Early iterations should address high-risk aspects to reduce overall project risk.

ITERATIVE & INCREMENTAL MODEL (unsummarized)

Project artifacts such as plans, specifications, and code evolve during each phase and over the life of the project. Artifacts are considered only when the software is released.

Incremental development is a scheduling and staging strategy that allows pieces of the system to be developed at different times or rates and integrated as they completed. Iterations need not be executed serially, but can be developed in parallel by separate teams of people.

Successive increments implement an increasing set of functionality and features until the final iteration, which produces the final product. Each iteration is a self-contained project with its own set of activities, plan, objectives, and measurable evaluation criteria.

The release is a working version of software that is either used internally by the project team or externally by stakeholders and customers.

Types of Releases

  • Proof of concept, or feasibility study, that is used to demonstrate or investigate feasibility of a particular aspect of the software. This includes producing software simulations.
  • A prototype that is a working version of software demonstrating a particular capability that is deemed high risk.
  • An internal release that is only used by the development team and is used to ensure that development is on track, elicit feedback, and provide a basis for further development and additional capabilities.
  • An external release that is shipped to customers for evaluation.

Testing each iteration as a self-contained project allows clear and manageable objectives to be set, and reduces overall project complexity by breaking it down into smaller pieces.

By producing software at the end of each iteration, project progress can more easily be monitored and planned, and feedback can be elicited to ensure that its capabilities are meeting stakeholder requirements.

Typically, early iterations generate software releases that address aspects of the project with the highest risk. In this way, as the project progresses, the overall project risk level is reduced.

SPIRAL MODEL

Overview

Barry Boehm's Spiral Model is one of the earliest and best-known iterative processes. It's called a "spiral" because development iterates as an outward spiral, with each cycle representing one iteration.

Risk-Driven Approach

The goal of each cycle is to increase system definition and implementation while decreasing risk. Risk management is built into the process with early risk identification in each iteration. The overall project risk reduces as you progress outward in the spiral.

Spiral Model Iteration Steps

  • Identify critical objectives and constraints
  • Evaluate project and process alternatives
  • Identify risks
  • Cost-effectively resolve risks using analysis, prototypes, etc.
  • Develop project deliverables
  • Plan for future cycles and update the project plan
  • Review with stakeholders and commit to proceeding

Implementation Pattern

Each spiral traversal produces incremental deliverables. Early iterations yield models or prototypes, while later iterations follow a more waterfall-like process producing more complete software versions. The number of cycles depends on project size, identified risks, and their retirement rate.

Advantages of the Spiral Model

  • Risks are managed early and throughout the process. Risks are reduced before they become problematic, as they are considered at all stages. As a result, stakeholders can better understand and react to risks.
  • Software evolves as the project progresses. It is a realistic approach to the development of large-scale software. Errors and unattractive alternatives are eliminated early.
  • Planning is built into the process. Each cycle includes a planning step to help monitor and keep a project on track.

Disadvantages of the Spiral Model

  • Complicated to use. Risk analysis requires highly specific expertise. There is inevitably some overlap between iterations.
  • May be overkill for small projects. The complication may not be necessary for smaller projects. It does not make sense if the cost of risk analysis is a major part of the overall project cost.

SPIRAL MODEL (unsummarized)

One of the earliest and best known iterative processes is Barry Boehm's Spiral Model. It is called a spiral because Boehm conceptualized development iterating as an outward spiral. It is a risk-driven process in which iterations have the specified goals.

A project starts at the center, as it were, and each cycle of the spiral represents one iteration. The goal of each cycle is to increase the degree of system definition and implementation, while decreasing the degree of risk.

Risk management is built into the process in that very early in each iteration, project risks are identified and analyzed and a plan for the iteration is created that includes mitigating some or all of the risks.

The overall project risk reduces the further along you are in the process. After the major risks are addressed and mitigated, the project transitions to a waterfall model as seen in the outer spiral.

Each iteration in Boehm's spiral model consists of the following steps:

  1. Identification of critical objectives and constraints of the product
  2. Evaluation of project and process alternatives for achieving the objectives
  3. Identification of risks
  4. Cost-effective resolution of a subset of risks using analysis, emulation, benchmarks, models, and prototypes
  5. Development of project deliverables including requirements, design, implementation, and testing
  6. Planning for next and future cycles - the overall project plan is updated, including schedule, cost, and number of remaining iterations
  7. Stakeholder review of iteration deliverables and their commitment to proceed based on their objectives being met

Each traversal of the spiral results in incremental deliverables. Early iterations produce either models, emulations, benchmarks, or prototypes, and later iterations follow a more waterfall-like process and incrementally produce more complete versions of the software.

Early iterations may exclude step 5, and later iterations may exclude step 4. The spiral model does not prescribe a set number of cycles but is dictated by the size of the project, the number of risks identified, and their rate of retirement.

AGILE MODEL

Origins

Started in 2001 when industry experts met to discuss improving software development. The group called themselves Agile Alliance and created the Manifesto for Agile Software Development (Agile Manifesto).

An agile software process is one that embraces and conforms to the Agile Manifesto values and principles.

Characteristics of Agile Processes

Agile processes are highly iterative and incremental, focusing on delivering working software quickly and responding to change effectively.

Common Agile Practices

  • Small, close-knit teams
  • Regular, frequent, disciplined customer requirements meetings
  • A code-centric approach, documentation on an as-needed basis
  • Customer representatives working within the team
  • The use of user stories as the basis for requirements
  • Pair programming, in which two programmers work at a single workstation
  • Continual unit-testing, and acceptance tests as means of getting customer expectations

Advantages of the Agile Process

  • The project always has demonstrable results. The end product of each iteration is working software.
  • Developers tend to be more motivated. Developers prefer to produce working artifacts and tend not to like creating documentation.
  • Customers are able to provide better requirements because they can see the evolving product.

Disadvantages of the Agile Process

  • Problematical for large applications. Agile methods are more steadily used for smaller projects. There is a debate about their utility for large projects.
  • Documentation output is questionable. Since documentation takes second place, there is a question as to whether necessary documentation will ever be produced.

PROTOTYPING

Purpose of Prototyping

Prototyping is an important risk management technique. It involves creating a partial implementation to identify and retire risky parts of a project.

Benefits

Prototyping helps obtain ideas about customer requirements and increases understanding of what's to come, saving expensive rework and removing future roadblocks.

Implementation Examples

Simple graphics showing GUIs using paint-type tools may be sufficient if the prototype's goal is to envision interfaces. More extensive prototypes can retire more risks.

When to Consider Prototyping

Consider prototyping when:

  • Requirements are unclear or complex
  • Stakeholders need visual confirmation
  • New technology is being implemented
  • High-risk components need validation early
